Hooks：条件反射机制

2026 · April Claude Code · Hooks · 自动化

上篇讲了 CLAUDE.md，给 cc 写一份规则说明书。

但说明书有个问题——它靠 cc "记着遵守"。你在 CLAUDE.md 里写"写完代码跑一下 lint"，有时候它会做，有时候……就没有然后了。就像跟实习生说"提交前检查一下格式"，第三周他还是忘了。说实话，被这件事坑过不止一次，每次都是 review 的时候才发现一堆格式问题，气昏。

Hooks 解决的就是这个问题。

不是让 cc 记着做，是去触发系统被动来做。

Hooks 是什么

一句话：在 cc 执行操作的前后，自动触发你的脚本。

CLAUDE.md 靠 cc 读懂并自觉执行，Hooks 是系统级别的自动触发，cc 没有选择权。

两者对比：

CLAUDE.mdHooks
本质给 cc 看的文字规则给系统执行的 Shell 脚本
执行方式cc 阅读后尽量遵守到时间点强制运行
可靠性偶尔会忘100% 触发
适合场景编码风格、项目背景格式化、安全拦截、通知

四个时机，覆盖全流程

Hooks 有五种事件类型，常用的四个：

— PreToolUse —：工具调用前触发。这是唯一能"拦截"操作的时机。cc 准备跑命令或写文件之前，你的脚本先跑一遍，exit 2 就是"不行，停下"。

— PostToolUse —：工具执行完成后触发。文件写完了，自动跑格式化；代码改完了，自动跑测试。善后工作全交给它。

— Notification —：cc 产生通知时触发。跑大任务时把 macOS 提醒挂上，不用盯着终端。

— Stop —：cc 完成一轮回复后触发。这轮问答结束了，播一个提示音。切出去做别的事，听到"叮"就知道好了。

整体流程大概长这样：你发消息 → cc 思考 → [PreToolUse] → 执行工具 → [PostToolUse] → 回复你 → [Stop]

三个即装即用的 Hook

① 完成提示音（30 秒上手）

跑大任务时最烦的事——不知道 cc 搞完了没有，得一直切回来看。

通过 Stop Hook 解决这个问题，在 ~/.claude/settings.json 里加：

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "afplay /System/Library/Sounds/Glass.aiff"
          }
        ]
      }
    ]
  }
}

macOS 自带的系统音效，不需要额外装任何东西。想换声音可以把 Glass.aiff 换成 Ping.aiff、Pop.aiff。

配完之后，你丢给 cc 一个任务，等听到玻璃音效就知道它搞完了。

② 自动格式化（最实用）

cc 写出来的代码，格式有时候跟项目规范对不上。不是说它态度不好，是这类机械性的东西本来就不该靠"记着"来保证——人也一样，我自己写代码心情好的时候也会跳过格式化，何况 AI。

PostToolUse 挂上 prettier，文件一落地就跑格式化，不给"尽量"留余地：

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_FILE_PATH\" 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}

matcher 里写 Write|Edit，匹配新建和修改两种操作。|| true 防止 prettier 报错把整个 Hook 流程中断。

从此 cc 写完文件，你看到的永远是格式化好的代码。

③ 保护关键文件（最安全）

有些文件不能动。package-lock.json、yarn.lock、数据库 migration 文件……cc 有时候心情好，会在你没注意时顺手"帮你"改一下。我有次回来看提交记录，lock 文件多了一堆 diff，很难受，查了半天才弄清楚是 cc 在"优化依赖"。

PreToolUse 加一层检查，exit 2 直接拦：

#!/bin/bash
input=$(cat -)
file_path=$(echo "$input" | jq -r '.tool_input.file_path // empty')

protected=("package-lock.json" "yarn.lock" "pnpm-lock.yaml" "migrations/")

for pattern in "${protected[@]}"; do
  if echo "$file_path" | grep -q "$pattern"; then
    echo "受保护文件，不允许修改: $file_path" >&2
    exit 2
  fi
done

exit 0

然后在 settings.json 里挂上：

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/protect_files.sh"
          }
        ]
      }
    ]
  }
}

cc 想改 lock 文件？不好意思，脚本 exit 2，操作直接被打回去，cc 会在回复里告诉你"该文件受保护，无法修改"。

怎么配置

方式 1：/hooks 交互式配置

在 cc 会话里输入 /hooks，会进入菜单界面，选事件类型、填命令、保存，不用手动碰 JSON。第一次用推荐这个，不容易写错格式。

方式 2：直接编辑 settings.json

配置文件有两个层级：

~/.claude/settings.json — 全局，所有项目生效

项目目录/.claude/settings.json — 项目级，只对当前项目生效，优先级更高

和 CLAUDE.md 的层级逻辑完全一样——全局是 baseline，项目可以 override。

一份可以直接复制的完整配置模板：

{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "afplay /System/Library/Sounds/Glass.aiff"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write \"$CLAUDE_FILE_PATH\" 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}

把这个丢到 ~/.claude/settings.json，提示音和自动格式化就都有了。

最后说一下 CLAUDE.md 和 Hooks 的配合，用一句话总结：

— CLAUDE.md 管意图，Hooks 管执行 ——

CLAUDE.md 里写"遵循 ESLint 规范"，是让 cc 生成代码时就往那个方向走。Hooks 里挂 eslint --fix，是写完之后兜底保障。一个告诉它应该怎么做，一个确保它最后真的做了。

两者不是替代关系，是同一件事的两个保险。

CLAUDE.md 是告诉螃蟹"应该怎么做"，Hooks 是让它"不得不这么做"。
